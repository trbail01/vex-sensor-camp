# Instructor Guide

## Materials List

- Pre-built VEX robots with claws
- Pressure, optical, and range sensors installed
- Laptops with the required VEX software installed
- Batteries, chargers, and spares
- Tape for floor lines and finish zones
- Small objects for pickup tasks
- Optional: Raspberry Pi, Home Assistant, Aqara sensors, smart bulbs, and smart switches

## Pre-Camp Setup Checklist

- Charge all robots and controllers
- Open the programming software on each laptop
- Test that every robot connects correctly
- Mark a driving lane, pickup zone, and finish line area
- Prepare a demo program for each mission
- Set out spare cables, batteries, and tape

## Robot Checklist

- Drive motors respond correctly
- Claw opens and closes
- Pressure sensor is firmly mounted and readable
- Range sensor faces forward near the claw
- Optical sensor points toward the floor
- Sensor cables are connected to the correct ports

## Laptop/Software Checklist

- Required software installed and updated
- Students can connect to the robot
- Live sensor values can be viewed or tested
- Code can be downloaded and run without extra setup

## Room Setup

- Use one programming station per team
- Leave a clear test area with room to drive forward
- Keep a simple demo lane near the front of the room
- Put a spare robot or teacher station in a visible location
- Store charging gear away from the test path

## Suggested Group Size

- Best: 2 students per robot
- Acceptable: 3 students per robot
- Helpful roles: driver, coder, tester

## Timing Advice

- Teach in short bursts of 5-10 minutes
- Move back to testing quickly
- Do not let teams stay stuck on a broken idea for too long
- Use the same language every time: input, logic, output

## Troubleshooting Tips

- If a sensor seems wrong, check the port and cable first
- If the robot misses objects, slow the drive speed before changing the whole program
- If the optical sensor acts strangely, test on the actual floor surface
- If students are guessing, have them read live sensor values first

## Backup Plans

- If one sensor fails, turn that mission into a teacher demo and let students use the other sensors
- If a robot stops working, move the team to a paper planning task while the robot is checked
- Keep printed pseudocode cards ready so students can keep working without hardware

## Middle School Delivery Tips

- Model each mission first
- Keep explanations concrete and brief
- Use lots of simple if/then language
- Celebrate progress, even if the first run is messy

## High School Delivery Tips

- Ask students to justify thresholds and explain their choices
- Encourage them to store values in variables
- Have them talk through debugging instead of just changing code randomly
- Push for a short explanation of why the routine works

## Optional Home Assistant / Zigbee Demo Notes

- Show a motion sensor or door sensor in Home Assistant
- Point out that the automation follows the same pattern as the robot: input, logic, output
- Use a smart bulb or smart switch so the result is easy to see
- Keep the demo short and clearly tied back to the robot missions
